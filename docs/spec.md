# 出退勤打刻アプリ 仕様書

## 1. 画面仕様

### 1-1. 全体レイアウト

スマホ縦持ち（375px 基準）の 1 ページ構成。
画面を上から「ヘッダー」「ステータスエリア」「打刻ボタンエリア」「課題報告エリア」「フッター」の 5 ブロックで構成する。

```
┌─────────────────────────┐
│  ヘッダー（会社名 / 日付）   │
├─────────────────────────┤
│  ステータスエリア            │
│  ・現在時刻（リアルタイム）   │
│  ・今日の出勤時刻           │
│  ・勤務時間（退勤後に表示）   │
├─────────────────────────┤
│  打刻ボタンエリア            │
│  ┌──────────┐ ┌──────────┐ │
│  │  出 勤   │ │  退 勤   │ │
│  └──────────┘ └──────────┘ │
├─────────────────────────┤
│  課題完了報告エリア          │
│  ・課題名テキストエリア       │
│  ・送信ボタン               │
├─────────────────────────┤
│  フッター（バージョン情報）    │
└─────────────────────────┘
```

---

### 1-2. ヘッダー

| 要素 | 内容 |
|------|------|
| タイトル | アプリ名（例: 勤怠システム） |
| 日付表示 | `YYYY年MM月DD日（曜日）` 形式 |

---

### 1-3. ステータスエリア

| 要素 | 表示条件 | 内容 |
|------|----------|------|
| 現在時刻 | 常時 | `HH:MM:SS` でリアルタイム更新（1秒ごと） |
| 出勤時刻 | 出勤打刻後 | `出勤: HH:MM` |
| 退勤時刻 | 退勤打刻後 | `退勤: HH:MM` |
| 勤務時間 | 退勤打刻後 | `勤務時間: H時間M分` |

---

### 1-4. 打刻ボタンエリア

#### 出勤ボタン
| 状態 | 表示 | 操作可否 |
|------|------|----------|
| 未打刻 | 緑色・有効 | 押せる |
| 打刻済み | グレー・無効 | 押せない |

#### 退勤ボタン
| 状態 | 表示 | 操作可否 |
|------|------|----------|
| 出勤前 | グレー・無効 | 押せない |
| 出勤後・退勤前 | 赤色・有効 | 押せる |
| 退勤済み | グレー・無効 | 押せない |

**ボタン押下時の動作フロー:**

```
ボタン押下
  └─ 確認ダイアログ（「出勤を打刻しますか？」）
       ├─ OK → GAS へ POST リクエスト送信
       │         ├─ 成功 → ステータス更新 + ボタン状態変更
       │         └─ 失敗 → エラートースト表示
       └─ キャンセル → 何もしない
```

---

### 1-5. 課題完了報告エリア

| 要素 | 仕様 |
|------|------|
| テキストエリア | 最大 200 文字、placeholder あり |
| 送信ボタン | テキストが空のとき無効化 |

**送信フロー:**

```
送信ボタン押下
  └─ 入力バリデーション（空欄チェック）
       ├─ NG → 「課題名を入力してください」と表示
       └─ OK → GAS へ POST リクエスト送信
                 ├─ 成功 → テキストエリアをクリア + 成功トースト
                 └─ 失敗 → エラートースト表示
```

---

### 1-6. トースト通知

画面下部に 3 秒間表示後に自動消去。

| 種類 | 色 | 用途 |
|------|----|------|
| success | 緑 | 打刻成功・報告送信成功 |
| error | 赤 | 通信エラー・バリデーションエラー |
| info | 青 | その他通知 |

## 2. データ構造

### 2-1. フロントエンド（localStorage）

アプリ起動時に当日の打刻状態を復元するためにローカル保存する。

```json
{
  "date": "2026-03-17",
  "clockIn": "09:00",
  "clockOut": "18:00"
}
```

| キー | 型 | 説明 |
|------|----|------|
| `date` | string | 打刻日（YYYY-MM-DD）。日付が変わったら自動リセット |
| `clockIn` | string \| null | 出勤時刻（HH:MM）。未打刻は null |
| `clockOut` | string \| null | 退勤時刻（HH:MM）。未打刻は null |

---

### 2-2. スプレッドシート

#### シート名: `勤怠`

| 列 | カラム名 | 型 | 例 |
|----|----------|----|----|
| A | 日付 | Date | 2026/03/17 |
| B | 社員番号 | string | EMP001 |
| C | 氏名 | string | 山田 太郎 |
| D | 出勤時刻 | string | 09:00 |
| E | 退勤時刻 | string | 18:00 |
| F | 勤務時間（分） | number | 540 |
| G | 備考 | string | （任意） |

#### シート名: `課題報告`

| 列 | カラム名 | 型 | 例 |
|----|----------|----|----|
| A | 日時 | DateTime | 2026/03/17 17:55 |
| B | 社員番号 | string | EMP001 |
| C | 氏名 | string | 山田 太郎 |
| D | 課題内容 | string | ○○機能の実装完了 |

---

### 2-3. フロントエンド → GAS へ送るリクエストボディ

#### 出退勤打刻

```json
{
  "action": "clockIn",
  "employeeId": "EMP001",
  "name": "山田 太郎",
  "timestamp": "2026-03-17T09:00:00+09:00"
}
```

`action` は `"clockIn"` または `"clockOut"`。

#### 課題完了報告

```json
{
  "action": "report",
  "employeeId": "EMP001",
  "name": "山田 太郎",
  "task": "○○機能の実装完了",
  "timestamp": "2026-03-17T17:55:00+09:00"
}
```

---

### 2-4. GAS → フロントエンドへ返すレスポンス

```json
{
  "status": "ok",
  "message": "出勤打刻を記録しました"
}
```

エラー時:

```json
{
  "status": "error",
  "message": "すでに出勤打刻済みです"
}
```

---

## 3. API 設計

GAS をウェブアプリとしてデプロイし、単一エンドポイントで `action` フィールドによって処理を分岐する。

### エンドポイント

```
POST https://script.google.com/macros/s/{DEPLOYMENT_ID}/exec
Content-Type: application/json
```

> **注意:** GAS の制約により GET/POST のみ対応。CORS 対応のため GAS 側で `ContentService` を使用する。

### アクション一覧

| action | 処理内容 | 使用シート |
|--------|----------|-----------|
| `clockIn` | 出勤打刻を記録 | 勤怠 |
| `clockOut` | 退勤打刻・勤務時間計算を記録 | 勤怠 |
| `report` | 課題完了報告を記録 | 課題報告 |

### レスポンス共通仕様

| フィールド | 型 | 説明 |
|------------|----|------|
| `status` | `"ok"` \| `"error"` | 処理結果 |
| `message` | string | ユーザー向けメッセージ |

---

## 4. GAS 関数一覧

### `Code.gs` — エントリーポイント

| 関数名 | 引数 | 戻り値 | 説明 |
|--------|------|--------|------|
| `doPost(e)` | `e: Event` | `ContentService` | POST リクエストを受け取り action で処理を振り分けるメインルーター |

```
doPost
  ├─ action === "clockIn"  → clockIn()  in sheet.gs
  ├─ action === "clockOut" → clockOut() in sheet.gs
  ├─ action === "report"   → saveReport() in sheet.gs
  └─ その他 → エラーレスポンス
```

---

### `sheet.gs` — スプレッドシート操作

| 関数名 | 引数 | 戻り値 | 説明 |
|--------|------|--------|------|
| `clockIn(data)` | `data: Object` | `Object` | 勤怠シートに出勤時刻を記録。重複チェックあり |
| `clockOut(data)` | `data: Object` | `Object` | 退勤時刻と勤務時間を記録。出勤記録の存在チェックあり |
| `saveReport(data)` | `data: Object` | `Object` | 課題報告シートに1行追記 |
| `findTodayRow(sheet, employeeId)` | `sheet, string` | `number \| null` | 当日・社員IDで行番号を検索するヘルパー |

---

### `line.gs` — LINE 通知

| 関数名 | 引数 | 戻り値 | 説明 |
|--------|------|--------|------|
| `sendLineNotify(message)` | `string` | `void` | LINE Messaging API でメッセージをプッシュ送信 |
| `buildClockInMessage(data)` | `data: Object` | `string` | 出勤通知メッセージ文字列を組み立てる |
| `buildClockOutMessage(data)` | `data: Object` | `string` | 退勤通知メッセージ文字列を組み立てる |
| `buildReportMessage(data)` | `data: Object` | `string` | 課題報告通知メッセージ文字列を組み立てる |

---

### `utils.gs` — ユーティリティ

| 関数名 | 引数 | 戻り値 | 説明 |
|--------|------|--------|------|
| `formatDate(date)` | `Date` | `string` | `YYYY/MM/DD` 形式の文字列を返す |
| `formatTime(date)` | `Date` | `string` | `HH:MM` 形式の文字列を返す |
| `calcWorkMinutes(clockIn, clockOut)` | `string, string` | `number` | 2つの `HH:MM` 文字列から勤務分数を計算する |
| `createResponse(status, message)` | `string, string` | `ContentService` | `{"status":..., "message":...}` の JSON レスポンスを生成する |

## 5. エラーケース

### 5-1. フロントエンドのバリデーションエラー

| ケース | 対処 |
|--------|------|
| 出勤打刻済みなのに再度出勤ボタンを押す | ボタン自体を無効化（操作不可） |
| 出勤前に退勤ボタンを押す | ボタン自体を無効化（操作不可） |
| 課題報告テキストが空欄のまま送信 | 「課題名を入力してください」をトーストで表示 |

### 5-2. GAS 側のビジネスロジックエラー

| ケース | GAS の返却 | フロント表示 |
|--------|------------|------------|
| 当日の出勤レコードが既に存在する | `status: "error"` | エラートースト |
| clockOut 時に出勤レコードが見つからない | `status: "error"` | エラートースト |
| 退勤時刻が出勤時刻より前になっている | `status: "error"` | エラートースト |

### 5-3. 通信エラー

| ケース | 対処 |
|--------|------|
| ネットワーク接続なし | 「通信に失敗しました。ネットワークを確認してください」をトーストで表示 |
| GAS が 5xx を返した | 「サーバーエラーが発生しました。時間をおいて再度お試しください」をトーストで表示 |
| GAS のタイムアウト（30秒超） | fetch に `AbortController` でタイムアウト設定し、エラートーストを表示 |

### 5-4. PWA / localStorage エラー

| ケース | 対処 |
|--------|------|
| localStorage が使えない（プライベートブラウジング等） | try-catch で握りつぶし、セッション中のみ状態を保持 |
| 日付をまたいでアプリを開いた | `date` フィールドと今日の日付を比較し、不一致なら localStorage をリセット |

## 6. 開発手順

### ステップ 1 — リポジトリと GitHub Pages の準備

1. GitHub にリポジトリを作成する
2. `main` ブランチを GitHub Pages のソースに設定する
3. フォルダ構成に従ってファイルを作成する（中身は空でよい）

### ステップ 2 — Google Spreadsheet の準備

1. 新規スプレッドシートを作成する
2. シート名を `勤怠` と `課題報告` に変更する
3. [2-2. スプレッドシート](#2-2-スプレッドシート) に従いヘッダー行を入力する
4. スプレッドシートの ID（URL の `/d/` と `/edit` の間の文字列）をメモする

### ステップ 3 — GAS のセットアップ

1. スプレッドシートのメニュー「拡張機能 > Apps Script」を開く
2. `gas/` フォルダ内の各ファイルをコピーして貼り付ける
3. `appsscript.json` の設定を反映する（マニフェストの編集を有効化してから貼り付ける）
4. スクリプトプロパティに以下を設定する:

| キー | 値 |
|------|----|
| `SPREADSHEET_ID` | ステップ 2 でメモした ID |
| `LINE_CHANNEL_ACCESS_TOKEN` | LINE Developers で発行したトークン |
| `LINE_USER_ID` | 通知先の LINE ユーザー ID |

5. 「デプロイ > 新しいデプロイ」でウェブアプリとして公開する
   - 実行者: 自分
   - アクセスできるユーザー: 全員
6. デプロイ URL をメモする

### ステップ 4 — LINE Messaging API の準備

1. [LINE Developers](https://developers.line.biz/) でチャネルを作成する（Messaging API）
2. チャネルアクセストークンを発行し、GAS スクリプトプロパティに設定する
3. 通知先ユーザーの LINE ユーザー ID を取得し、スクリプトプロパティに設定する

### ステップ 5 — フロントエンド実装

実装順序:

1. `index.html` — 画面の骨格を作る
2. `style.css` — スマホ向けのスタイルを当てる
3. `app.js` — 以下の順で実装する
   1. 現在時刻のリアルタイム更新
   2. localStorage による状態管理（保存・読み込み・リセット）
   3. ボタンの有効/無効切り替えロジック
   4. GAS への fetch 送信処理
   5. トースト通知の表示/非表示
   6. 課題報告フォームのバリデーションと送信
4. `manifest.webmanifest` — PWA のメタ情報を記述する
5. `sw.js` — Service Worker でオフライン対応（静的ファイルのキャッシュ）を実装する

### ステップ 6 — GAS 実装

実装順序:

1. `utils.gs` — 日付・時刻フォーマット、レスポンス生成などのユーティリティから実装する
2. `sheet.gs` — スプレッドシートへの読み書きロジックを実装する
3. `line.gs` — LINE 通知ロジックを実装する
4. `Code.gs` — ルーターを実装し、上記関数を呼び出す

### ステップ 7 — 結合テストと公開

1. `docs/test-checklist.md` に従って動作確認する
2. エラーケースを意図的に再現して確認する
3. GitHub に push して GitHub Pages で表示確認する
4. スマホ実機でホーム画面に追加し PWA として動作確認する
